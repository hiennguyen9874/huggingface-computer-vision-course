Kỹ thuật 01 · Pruning

Loại bỏ phần ít quan trọng

Một số weight, neuron, channel hoặc filter gần như không ảnh hưởng tới output.

  • Model nhỏ hơn, ít phép tính hơn
  • Prune quá mạnh có thể làm giảm accuracy
  • Ví dụ: 0.0001 → 0
Hai hướng

Unstructured

Xóa từng weight. Sparsity cao, ít ảnh hưởng accuracy hơn — nhưng hardware chưa chắc tăng tốc.

Structured

Xóa channel/filter/layer/block. Dễ tăng tốc thực tế trên phần cứng phổ thông.

Kỹ thuật 02 · Quantization

Ít bit hơn, cùng mục tiêu inference

FP32 → FP16 → INT8

FP32 dùng 32 bit; INT8 dùng 8 bit. Nếu phần cứng hỗ trợ tốt, model nhỏ hơn, nhanh hơn và tiết kiệm điện hơn.

Con số cần nhớ
4×

Giảm kích thước lý thuyết khi chuyển từ FP32 sang INT8:

32 bit / 8 bit = 4

Rủi ro: accuracy giảm, một số layer nhạy cảm, cần calibration hoặc QAT.

Post-training quantization

Convert sau khi train — dễ bắt đầu, cần kiểm tra accuracy

quantize.py
import torch

model = ...  # model FP32 đã train
model.eval()

quantized_model = torch.quantization.quantize_dynamic(
    model,
    {torch.nn.Linear},
    dtype=torch.qint8
)

CNN/Transformer thực tế cần chọn backend và calibration phù hợp

Ba hướng bổ sung

Khi quantization chưa đủ

  • Knowledge distillation: teacher lớn truyền “soft labels” cho student nhỏ; loss gồm hard loss + soft loss, temperature và alpha.
  • Low-rank approximation: xấp xỉ W(m×n) ≈ A(m×k) × B(k×n), với k nhỏ hơn.
  • Hardware-specific: chọn TensorRT, OpenVINO, Edge TPU compiler hoặc Optimum theo target.

Serialization + runtime

ONNX tách training framework khỏi serving runtime

export_and_run.py
import torch
import onnxruntime as ort

model.eval()
x = torch.randn(1, 3, 224, 224)
torch.onnx.export(model, x, "model.onnx",
  input_names=["input"], output_names=["output"], opset_version=17)

session = ort.InferenceSession("model.onnx")
name = session.get_inputs()[0].name
outputs = session.run(None, {name: x.numpy()})

05

Deployment platform

Cloud, edge hay mobile?

Vị trí của inference quyết định architecture

Cloud

Scale và quản trị tập trung

  • AWS, Google Cloud, Azure hoặc server riêng
  • Nhiều CPU/GPU/RAM; dễ logging, monitoring, CI/CD
  • Phù hợp model lớn, API, batch inference
  • Đổi lại: cost, network latency, privacy, internet dependency
Edge + mobile

Đưa inference gần dữ liệu

  • Camera AI, Raspberry Pi, Jetson, IoT, factory edge
  • Latency thấp, offline, privacy tốt hơn
  • Mobile: Core ML, TFLite, ONNX Runtime Mobile
  • Đổi lại: tài nguyên hạn chế, update khó, phụ thuộc hardware

Hạ tầng production

Khi nào cần orchestration?

  • Kubernetes: scale replica model server, rolling update, rollback, health check, load balancing, CPU/GPU resource management.
  • Chỉ nên dùng khi có nhiều service, cần scale linh hoạt và team có DevOps/MLOps.
  • Hugging Face Inference Endpoints: managed deployment cho Transformers, Diffusers hoặc custom model.
  • Phù hợp khi muốn deploy nhanh mà không tự quản lý container/GPU — nhưng phải cân nhắc cost và vendor lock-in.

Transport

REST, gRPC hay managed service?

  • HTTP REST: dễ dùng, phổ biến, debug tốt
  • gRPC: contract rõ qua protobuf, service-to-service, streaming; phức tạp hơn
  • Cloud managed: scale nhanh, ít vận hành; đổi lại cost và vendor lock-in
Computer Vision postprocess

Output còn phải được giải mã

  • Classification: softmax nếu cần xác suất; sigmoid cho multi-label
  • Detection: decode boxes + NMS nếu model yêu cầu
  • Segmentation: resize mask về ảnh gốc
  • Pose: convert keypoint coordinates

Monitor liên tục: latency, throughput, errors, confidence, input statistics, output distribution, CPU/GPU, memory và drift signal.

Best practices

ML production là một hệ thống sống

  • MLOps: version code, dataset, model; CI/CD; registry; monitoring; rollback; retraining.
  • Load testing: mô phỏng tải thật, đo P95/P99, throughput, resource, error rate.
  • Anomaly detection: phát hiện input/output bất thường và distribution shift.
  • Security & privacy: TLS, access control, không log dữ liệu nhạy cảm, bảo vệ endpoint/model.
Release safely

Canary trước, full traffic sau

  • Canary / A/B: route một phần nhỏ traffic sang model mới
  • So sánh accuracy proxy, engagement, latency, errors, business metric
  • Tăng traffic dần khi model mới ổn định
  • Có rollback nhanh nếu model mới gây lỗi
Continuous evaluation

Model xuống cấp theo thời gian

Camera, sensor, người dùng, môi trường đều đổi. Cần đánh giá liên tục, retrain khi cần và ghi rõ owner, giới hạn, dependency, hardware, ngày deploy.

observe → compare → decide → rollback / retrain
